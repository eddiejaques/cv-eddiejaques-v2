01 — The Business Driver

Three products, no shared way to measure them

Leadership needed basic growth numbers — daily and monthly active users, retention, feature usage — across all three products, but no tracking setup existed to produce them.

GAP

No usage metrics, anywhere

None of the three products had a working setup to measure how many people used them, or how often.

No usage data = leadership can't see what's working
BUDGET

Paid analytics tools were off the table

The startup's budget required staying on Mixpanel's free plan, which limits how much data and detail you can track.

Free plan only = tracking has to be designed around real limits
SCALE

One setup had to work for three products

Whatever was built needed to give a combined view across Typesy, Spreeder, and Wordela, not three separate setups.

3 separate products = no easy way to compare them
02 — The Obstacles

What a full site audit turned up

Before designing anything, I audited the live products end to end. That audit surfaced three problems standing in the way of clean tracking.

DATA

User actions were hidden behind ID codes

The free plan blocked access to lookup tables, so events only showed raw codes like course_id or game_id — not anything a person could read.

Raw ID codes = reports nobody can read or trust
TRACKING

The apps didn't change pages when used

The core features (course player, game player, word lists) load content without changing the page URL, so standard page-view tracking couldn't see people dropping off mid-task.

No page changes = drop-off points stay invisible
QUALITY

Each team was logging data differently

Early tracking attempts across the three products used inconsistent names and formats, risking duplicate or conflicting data once combined.

Inconsistent logging = combined reports don't add up
03 — The Solution

A tracking design built around the free plan's limits

Working directly with leadership and engineering, I designed a tracking setup that worked around each limitation, then helped build and check it before launch.

1
Define the KPIs with leadership CXO interviews → roadmap

Worked directly with the startup's leadership team to agree on the core numbers that mattered — active users, retention, and feature engagement — and turned that into a tracking roadmap.Stakeholder alignment

2
Attach context to every event automatically

Set up a small set of shared details (which product, what type of user, which company) to be attached automatically to every action a user takes, so reports could be filtered and grouped without needing the paid lookup tables.Shared event context

3
Track in-app steps directly

Built custom tracking for the course player, game player, and word-list tools so each step a user takes inside these tools is recorded directly, even though the page itself never changes.In-app step tracking

4
Write the naming rules, then test live

Wrote a simple naming guide (consistent event and property names) for engineering to follow, then worked alongside engineers to fix bugs and check real, live data before anything went to production.Naming guide · QA
