01 — The Business Driver

A fully automated pipeline with one manual step left in it

career-ops handled offer evaluation and CV generation, but cover letters required the user to write from scratch — or accept generic AI output. Both paths undermined the pipeline's value.

GAP

No path from job description to approved PDF

The pipeline had no structured way to go from a job description to a finished cover letter. Users fell back to manual writing or copied boilerplate.

Manual gap = cover letters disconnected from the rest of the pipeline
QUALITY

Generic letters and over-automated ones both fail

Copy-paste templates are spotted immediately. AI-drafted letters that stuff keywords or invent metrics are worse — they break trust with the interviewer.

AI hallucinations = invented claims that collapse under interview questioning
CONTROL

No approval gate before submission

Without an explicit sign-off step, auto-generated content could reach employers before the user had read it, creating risk on every application.

No review gate = application sent on AI output, not user intent
02 — The Obstacles

Three constraints that couldn't be traded away

Solving the gap meant navigating content integrity, repo architecture, and maintainer standards — any one of which could have blocked the PR.

INTEGRITY

AI models paraphrase metrics — and get them wrong

Standard AI drafting extrapolates or invents numbers from CV bullet points. Every invented metric is a claim the user can't defend at interview.

Extrapolated metric = candidate flagged as untrustworthy on first question
ARCHITECTURE

The repo is Node.js-only — no new runtimes

An early draft used a Python helper for PDF generation. The maintainer's explicit standard: zero new deps, one runtime. That draft would have been rejected.

New runtime = PR blocked before any feature review
SCOPE

Two entry points, one coherent flow

Letters had to work from a standalone JD paste and from the existing offer evaluation report. Both paths needed to produce the same quality output without duplicating logic.

Duplicated logic = two flows to maintain, guaranteed to diverge
03 — The Solution

One flow, one gate, one output

The design principle was simple: the PDF is the last thing produced, not the first. Everything before it is research, selection, and approval.

1
10-step conversational flow JD → research → prompts → draft → approval

A mode file drives the letter from JD input through company research (three web searches), keyword extraction (ATS-critical vs. human trust signals), four mandatory prompts the user must answer before any draft is written, and an explicit approval gate before the PDF is produced. "Just generate it" has no effect — the gate cannot be bypassed.modes/cover.md · Claude Code

2
Achievement selection from cv.md score → select → preserve

Every bullet point in cv.md is scored against the JD's top competencies. The 4–5 highest-scoring bullets are selected, with at least one metric per bullet. Exact wording from the CV is preserved — keyword mirroring shifts vocabulary around the bullets, never the metrics themselves.Achievement scoring · cv.md

3
Node.js PDF renderer JSON payload → HTML template → A4 PDF

A Node.js ESM script accepts a structured JSON payload, performs HTML-escaped token replacement into a print-optimized A4 template, then renders to PDF via the same Playwright pipeline used for CV generation. No new runtime, no new dependency — the PR was mergeable from day one.generate-cover-letter.mjs · Playwright

4
10 language rules baked in active voice · no buzzwords · every claim grounded

Ten rules enforced at the drafting step: active voice only, a hard-banned buzzword list (leverage, synergy, spearheaded, actionable insights, north star), no filler openers, every claim backed by a number or a named system, and a 350–420 word body target. Rules apply on every letter and cannot be turned off.Language rules · 350–420 word target

05 — What the Feature Makes Possible

Tailored, approved letters in minutes — not hours

⚡

Every letter is company-specific and human-approved before it leaves the pipeline

The approval gate means the user reads the full draft in chat and says yes before any PDF exists. That single constraint separates this from every AI cover letter tool: no letter goes to an employer the user hasn't read and signed off.

⊕

ATS targeting without keyword stuffing

Keywords from the job description are classified as ATS-critical or human trust signals. Mirroring applies to vocabulary around the bullet points — never to the metrics or the claims themselves.

✕

Zero new infrastructure added to the repo

Routing PDF generation through the existing Playwright pipeline instead of adding a Python helper kept the repo at one runtime. The maintainer called it out explicitly: "zero new deps is exactly the repo's way."

↻

Works for any user, any role, any country

The renderer reads all identity, credentials, and proof points from cv.md and profile.yml at runtime. No names, metrics, or sentences are hardcoded — the same script generates a letter for a DevOps engineer in Tokyo and a Director of Data in Munich.

⤳

Consistent with the CV — one application packet

The cover letter uses the same A4 layout, typography, and Playwright renderer as the CV output. Both documents arrive as a matched pair, not a cover letter from one tool and a CV from another.
