Case Study · Tag Management Migration

Migrating 900 tags across 40+ domains with zero reporting disruption.

Optum's Dynamic Tag Manager was end-of-life with no third-party plugin support, but every insight and report had to keep running through the cutover. I planned and led the migration to Adobe Experience Platform Launch end to end.

Role Led the tag management migration — As-Is audit, phased migration plan, go-live, and governance recommendations

2018-2020 900 rules, data elements & tags 40+ domains 3-4 months · Feb-May '19 0 reporting disruption
01 — The Business Driver

The legacy tag manager was running out of road

Optum's Dynamic Tag Manager was approaching end-of-life with no vendor support, forcing a migration to Adobe Experience Platform Launch before it broke.

END-OF-LIFE

No support for the old tool

Adobe's Dynamic Tag Manager was being sunset, leaving Optum on an unsupported platform for core data collection.

Vendor support ending = core data collection at risk
ECOSYSTEM

No third-party plugin support

The legacy tool couldn't integrate with the broader martech stack, limiting what the tag layer could do.

No plugin ecosystem = martech stack stuck in place
CONTINUITY

Reporting couldn't break

The migration had to preserve the insight-generation and reporting pipelines that the business depended on daily.

Daily reporting depends on this layer = no room for downtime
02 — The Obstacles

What made this more than a lift-and-shift

A straightforward tool swap on paper became a full re-platforming effort once scale, missing documentation, and governance gaps surfaced.

SCALE

900 items across 40+ domains

Every rule, data element, and tag had to be inventoried and migrated individually across the full domain footprint.

900 items, 40+ domains = no shortcut to a full re-platform
DOCUMENTATION

Missing documentation

Much of the existing tag configuration had no documentation, so current state had to be reconstructed before migration could be planned.

No documentation = current state had to be reverse-engineered first
GOVERNANCE

Weak data governance

Underlying data governance gaps meant the migration also had to surface and address quality issues, not just relocate tags.

Weak data governance = quality issues hidden inside the migration
03 — The Solution

An As-Is audit, phased migration, and governed go-live

The migration was sequenced to document current state first, then move 900 items with a go-live plan designed to minimize data loss.

1
As-Is documentation current state → baseline

Created an As-Is plan documenting the current Dynamic Tag Manager configuration across all 40+ domains as the migration baseline.Audit · documentation

2
Phased migration plan

Drafted a project plan breaking the 900-item migration into phases across the 3-4 month window.Project planning

3
Go-live with loss prevention

Introduced a go-live plan designed to minimize data loss during cutover from Dynamic Tag Manager to Adobe Launch.Adobe Launch · cutover

4
Governance recommendations

Submitted recommendations to improve data governance, addressing the gaps surfaced during the migration.Data governance
